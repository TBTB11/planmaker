# 学習計画ツール_AI駆動開発ドキュメント計画

学習計画管理ツール

AI駆動開発ドキュメント計画

作成日: 2026/2/13

1. 概要

本書は、AI駆動開発を成功させるために必要なドキュメントを整理し、その作成優先度を定めるものである。

💡 AIにコードを生成させるには、構造化された仕様が重要です。

ドキュメントは以下の3つの優先度に分類される：

🔵 必須（Phase 1開始前）

🟣 推奨（Phase 2以降）

🟢 任意（品質向上）

2. 必須ドキュメント（Phase 1開始前に必要）

開発を開始する前に必ず作成するドキュメント。これらがないとAIは正確なコードを生成できない。

ドキュメント

内容

AI開発での目的

データモデル定義

エンティティ、属性、型、関係性をJSON Schemaで定義

AIがデータ構造を正確に理解してコード生成

ユーザーストーリー集

具体的な操作シナリオを「〇〇として△△したい」形式で記述

AIが実装の目的を理解

画面遷移図

画面間の遷移とトリガーを図示

AIがルーティングやナビゲーションを生成

技術スタック選定書

フレームワーク、ライブラリ、ツールの選定と理由

AIが適切なコードを生成

2.1 データモデル定義

項目

詳細

形式

JSON Schema

含める内容

エンティティ一覧、各属性の型・制約、エンティティ間の関係性、必須/任意の区分

AIへの効果

型安全なコード生成、バリデーションロジック自動生成、データベース設計の基礎

2.2 ユーザーストーリー集

項目

詳細

形式

「〇〇として、△△したい。なぜなら□□だから」形式

含める内容

アクター、目的、理由、受け入れ基準

AIへの効果

機能の目的理解、エッジケースの把握、テストケース生成の基礎

2.3 画面遷移図

項目

詳細

形式

フローチャートまたはMermaid記法

含める内容

画面一覧、遷移のトリガー（ボタン押下、条件分岐等）、戻る操作の可否

AIへの効果

ルーティング設計、ナビゲーションコンポーネント生成、画面コンポーネントの分割

2.4 技術スタック選定書

項目

詳細

形式

選定結果＋選定理由の文書

含める内容

フレームワーク、状態管理、UIライブラリ、ビルドツール、テストツール

AIへの効果

ベストプラクティスに沿ったコード生成、一貫した技術スタイル

3. 推奨ドキュメント（Phase 2以降で段階的に作成）

開発と並行して作成するドキュメント。品質と開発効率を向上させる。

ドキュメント

内容

AI開発での目的

UIコンポーネント一覧

再利用可能なUIコンポーネントの定義

AIが一貫したUI実装を生成

単元マスターデータ

学年×科目ごとの標準単元リスト（JSON形式）

一般情報ベース補完の基礎データ

AIプロンプト設計書

各補完機能のプロンプトテンプレート

AI補完機能の実装

API設計書

エンドポイント、リクエスト/レスポンス形式

バックエンド実装時（必要な場合）

3.1 UIコンポーネント一覧

項目

詳細

形式

コンポーネント名、Props定義、使用例

含める内容

ボタン、入力フォーム、カード、テーブル、モーダル等の共通コンポーネント

AIへの効果

一貫したUIデザイン、コンポーネントの再利用、スタイルの統一

作成タイミング

Phase 1のコア機能開発と並行して

3.2 単元マスターデータ

項目

詳細

形式

JSON形式の階層構造データ

含める内容

学年、科目、単元名、標準的な順序、難易度目安

AIへの効果

一般情報ベースのAI補完機能の実現

作成タイミング

AI基盤（Phase 2）開発前に

3.3 AIプロンプト設計書

項目

詳細

形式

機能別のプロンプトテンプレート集

含める内容

入力変数、期待する出力形式、Few-shot例、制約条件

AIへの効果

AI補完機能の安定した動作、出力品質の向上

作成タイミング

AI基盤（Phase 2, 5, 7）開発時

3.4 API設計書

項目

詳細

形式

OpenAPI (Swagger) 形式

含める内容

エンドポイント、HTTPメソッド、リクエスト/レスポンススキーマ、エラーコード

AIへの効果

フロントエンドとバックエンドの整合性確保

作成タイミング

バックエンドが必要になった場合

4. 任意ドキュメント（品質向上）

品質を高めるためのドキュメント。必須ではないが、作成することで堅牢性と保守性が向上する。

ドキュメント

内容

AI開発での目的

テストケース

機能ごとのテストシナリオ

AIがテストコードを生成

エラーハンドリング方針

エラー種別と対応方法

堅牢な実装

アクセシビリティ要件

対応すべきアクセシビリティ基準

包括的なUI

4.1 テストケース

項目

詳細

形式

Given-When-Then形式またはテスト表

含める内容

正常系、異常系、境界値、エッジケース

AIへの効果

テストコードの自動生成、品質保証の効率化

作成タイミング

各機能の実装後

4.2 エラーハンドリング方針

項目

詳細

形式

エラー分類表＋対応方針

含める内容

エラー種別、エラーコード、ユーザー向けメッセージ、リカバリー方法

AIへの効果

一貫したエラー処理、ユーザーフレンドリーなエラー表示

作成タイミング

Phase 2以降、実装中に追記

4.3 アクセシビリティ要件

項目

詳細

形式

WCAG基準に基づくチェックリスト

含める内容

対応レベル（A/AA/AAA）、具体的な実装要件、検証方法

AIへの効果

アクセシブルなHTML生成、ARIA属性の適切な使用

作成タイミング

UI改善フェーズ

7. サマリー

優先度

ドキュメント数

主な内容

🔵 必須

4件

データモデル、ユーザーストーリー、画面遷移図、技術スタック

🟣 推奨

4件

UIコンポーネント、単元マスター、AIプロンプト、API設計

🟢 任意

3件

テストケース、エラーハンドリング、アクセシビリティ

✅ まず必須ドキュメント4件を作成し、開発を開始する。
✅ 推奨ドキュメントは開発と並行して段階的に作成する。
✅ 任意ドキュメントは品質向上が必要になった時点で作成する。
